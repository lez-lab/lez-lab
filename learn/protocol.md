# LEZ Protocol Overview

## What LEZ Is

The Logos Execution Zone (LEZ) is the execution layer of the Logos network. It is a Layer 2 blockchain built on top of the Nomos data availability and consensus layer. LEZ handles program execution, state management, and transaction processing. Its distinguishing feature is native support for both public and private state within a single protocol, using zero-knowledge proofs (ZKPs) generated by the Risc0 zkVM.

LEZ is sometimes referred to as LSSA (Logos Sequence/State Architecture) or NSSA in the codebase. These names describe the same system at different levels of abstraction. LSSA is the specification; NSSA is the Rust implementation; LEZ is the user-facing name for the execution zone as a whole.

## Where LEZ Sits in the Logos Stack

```
Logos Application Layer (logos-core, wallet UI, modules)
    ↓
LEZ Execution Zone (NSSA — programs, state, transactions)
    ↓
Nomos Layer 1 (data availability, consensus via Cryptarchia)
```

The Logos application layer communicates with LEZ through a wallet FFI library (`wallet_ffi`), which is wrapped by a Qt plugin (`logos-execution-zone-module`) for integration with the Logos desktop application. Users can also interact directly with the sequencer via the NSSA API.

## State

The protocol maintains three categories of state:

```rust
V02State {
    public_state: Map<AccountId, Account>,
    private_state: (CommitmentSet, NullifierSet),
    programs: Map<ProgramId, Program>,
}
```

**Public state** is a key-value map from AccountId to Account. Conceptually, every possible AccountId maps to an Account. In practice, only accounts with non-default values are stored. Accounts not explicitly in the map are treated as having the default values (zero balance, no data, default program owner, zero nonce).

**Private state** consists of two data structures. The CommitmentSet is a Merkle tree of all active private account commitments. It supports membership proof generation and verification. The NullifierSet is a set of all nullifiers that have been revealed, used to prevent double-spending of private commitments.

**Programs** is a map from ProgramId to compiled program bytecode (Risc0 R0BF format). Programs are loaded at genesis (built-in programs) or deployed via program deployment transactions.

## Account Model

All accounts share a common structure:

```rust
struct Account {
    program_owner: ProgramId,   // which program controls this account
    balance: u128,              // native token balance
    data: Data,                 // arbitrary bytes (max 100KB), interpreted by program_owner
    nonce: u128,                // replay protection (public) or blinding factor (private)
}
```

Public accounts are identified by `AccountId = SHA256(prefix || public_key)` where the public key is a secp256k1 key used for BIP-340 Schnorr signatures.

Private accounts are identified by `AccountId = SHA256(prefix || npk)` where npk is the nullifier public key derived from the nullifier secret key (nsk).

Programs can also own accounts via Program Derived Addresses (PDAs), computed deterministically from the program ID and a seed value.

## Programs

Programs define the logic for operating on accounts. They are stateless. A program takes a list of accounts with metadata and an instruction data payload, and outputs updated account states plus optional chained calls to other programs.

```rust
type Program = fn(
    Vec<AccountWithMetadata>, InstructionData
) -> (Vec<AccountPostState>, Vec<ChainedCall>);
```

All programs must satisfy these invariants:

1. Input and output account lists have the same length with unique AccountIds.
2. Programs cannot modify the nonce or program_owner fields.
3. Only the owning program can decrease an account's balance or change its data.
4. Total balance across all accounts is conserved (checked with u256 arithmetic to prevent overflow).
5. Accounts with default program_owner after execution must have been default before execution.

### Built-in Programs (v0.3)

- **Authorized Transfer**: Moves native tokens between accounts. Requires sender authorization.
- **Piñata**: Distributes tokens to accounts that solve a challenge. Used for testnet faucet functionality.
- **Token Program**: Handles user-defined fungible tokens. Supports token definition, holding accounts, transfers, minting, and burning.
- **AMM Program**: Automated market maker for token swaps. Supports pool initialization, liquidity management, and constant-product swaps. Uses chained calls to the Token Program for vault operations.

### Chained Calls

A program can invoke another program through chained calls. Each call specifies the target program ID, instruction data, account pre-states, and PDA seeds for authorization. The current limit is 10 chained calls per transaction. Chained calls enable composability: the AMM program, for example, uses chained calls to the Token Program for all token operations.

### Account Claiming

Programs take ownership of uninitialized accounts through the claiming mechanism. When a program outputs an `AccountPostState` with `claim = true`, the protocol sets the account's `program_owner` to the executing program's ID. This only works for accounts that currently have the default program owner.

## Transaction Types

LEZ v0.3 supports three transaction types.

### Public Transaction

Used for operations on public accounts. Contains a message (program ID, account IDs, nonces, instruction data) and a witness set (BIP-340 Schnorr signatures).

The sequencer validates signatures, checks nonces against current state, executes the program, verifies program behavior invariants, and applies the state diff.

### Privacy-Preserving Transaction

Used for operations involving private accounts, including transfers between public and private state (shielding/deshielding) and fully private transfers.

Contains a message (public account IDs and post-states, encrypted private post-states, new commitments, new nullifiers) and a witness set (a Risc0 ZK proof plus any needed signatures for public accounts).

The program is executed off-chain by the user. The privacy-preserving circuit wraps the program proof and handles commitment computation, nullifier derivation, and encryption. The sequencer only verifies the proof and applies the state transition.

### Program Deployment Transaction

Deploys a new program to the chain. Contains the compiled program bytecode. The program ID is derived deterministically from the bytecode. A program can only be deployed once — if the ID already exists in state, the deployment is rejected.

## The Sequencer

The sequencer is the entity that orders transactions into blocks and maintains the canonical state. In LSSA v0.3, the sequencer model is centralized (single sequencer with signature authority). Future versions will move toward decentralized sequencing with proof-of-leadership.

The sequencer:

1. Receives transactions from the mempool
2. Validates each transaction against the current state
3. Executes public transactions and verifies proofs for privacy-preserving transactions
4. Applies valid state transitions
5. Produces blocks containing the ordered transactions

## Block Finality and State

Blocks in LEZ are produced by the sequencer and anchored to the Nomos L1 for data availability and finality. Each block contains an ordered list of transactions. State transitions are deterministic: given the same starting state and the same block, any node will arrive at the same resulting state.

The state after each block reflects all committed transactions:

- Public accounts updated according to program outputs
- New commitments inserted into the CommitmentSet Merkle tree
- Revealed nullifiers added to the NullifierSet
- Signer nonces incremented
- Any newly deployed programs added to the programs map

The CommitmentSet maintains a root history (set of all past Merkle roots). This allows privacy-preserving transactions to reference a slightly stale root during proof generation without being invalidated by concurrent transactions that modify the tree.
